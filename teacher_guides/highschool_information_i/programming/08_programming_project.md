# 教師用ガイド: テストを伴うプログラミング・プロジェクト

## レッスンのねらい

星見高校メディアラボの架空の印刷キュー報告ツールを、5〜6時限で実現可能な中核へ絞ります。中核は到着時刻が非減少の入力契約、同時到着での供給順保持、受付順（FIFO）、供給された関数シグネチャ、段階的なチェックポイント、通常・同時到着・降順・境界・無効・空のテストです。

- C9.O1: 限定されたFIFOの問題を分解し、Pythonへ実装する。
- C9.O2: 通常、同時到着、降順、境界、無効、空のテストを実行する。
- C9.O3: R1〜R5に照らして証拠、限界、改善、再テストを説明する。

## 前提確認

C6の関数とリスト行、C7の手順・トレース・テスト、C8の仮定と限界を確認します。FIFOの入力契約は本課題で定義し、「実行できる」「期待値と一致する」「要件を満たす」は別の確認であることを短い例で分類させます。

## 課題の境界

最大10件、一台、固定速度、FIFO、固定された合成データという範囲を維持します。GUI、ファイル保存、ネットワーク、実機制御、認証、実在利用者の調査、両面印刷へ広げません。

依頼は `[id, arrival, sheets]` のリスト行です。到着時刻は非減少順で、同時到着は供給順を保ちます。途中で到着時刻が小さくなる入力列は全体を無効とします。辞書の項目不足を検査する要件はありません。
中核で必要な関数名と引数、空の出力リスト、結果行の形、理由文字列、比較用 `show_check` は本文で供給します。
`append` はPython固有の部品として短く教えますが、条件、計算、繰り返し、関数の接続は学習者が実装します。

## 学習者用本文に沿った進め方

1. R1〜R5の入力、期待する動作、確認方法を読み合わせる。
2. FIFOの自然言語手順と、各関数の責務、入力、戻り値を書く。
3. 供給されたシグネチャと `[id, arrival, sheets]` の添字を確認する。
4. 一件の検査を通常・無効入力で完成させる。
5. 有効な依頼だけを残し、通常順と同時到着で供給順が変わらず、降順入力が無効になることを確認する。
6. 有効な2件から開始・終了・待ち時間を手計算と照合する。
7. 全件、平均、空入力へ段階的に広げる。
8. 通常、同時到着、降順、境界、無効、空の期待値を実行前に書き、実際値と比べる。
9. R1〜R5を証拠付きで評価し、限界と一つの改善・再テストをまとめる。

## 5〜6時限の時間配分

中核は250〜300分を目安にします。学習者の追跡とデバッグの状況に応じて組み替える柔軟な時間帯であり、固定時間ではありません。

- 第1時: 要件、関数設計、一件の検査。
- 第2時: 有効な依頼の抽出とFIFO順。
- 第3時: 開始、終了、待ち時間。
- 第4時: 平均と通常・境界・無効・空のテスト。
- 第5時: 件数上限を含む統合テストと修正。
- 第6時: 要件別評価、限界、改善、再テスト、提出物点検。

5時限の場合は、評価メモを各時限の最後に少しずつ書かせ、第6時の改善を一つの小さな再テストへ絞ります。短い印刷優先、GUI、追加データ構造は中核時間へ入れません。

## 最小実行可能な中間確認

### チェックポイント1: 検査

- `validation_error(0, 1)`、`validation_error(0, 0)`、`validation_error(-1, 5)` の期待値と実際値があるか。
- 複数の条件を一度に直さず、どの入力で失敗したか特定できるか。

### チェックポイント2: FIFO

- 無効な `M514` を除いているか。
- `M511`、`M512`、`M513`、`M515` の順序を変えていないか。
- 同時到着を供給順のまま処理し、降順入力を `到着時刻は非減少順` として全体無効にしているか。
- 関数が固定一覧を隠れて参照せず、引数を使っているか。

### チェックポイント3: 時刻

- 有効な2件について、開始、終了、待ち時間を手計算と照合したか。
- 有効な入力の全結果行で待ち時間が0以上か。
- プリンターが空く時刻を一件ごとに更新しているか。
- 結果のリスト行で各添字の意味が決まっているか。

### チェックポイント4: 平均と空入力

- 待ち時間の合計と件数をループで求めているか。
- 件数0で割り算をしていないか。
- R5「空の入力でも異常終了せず、空の結果を返す」の実行証拠があるか。

### チェックポイント5: 統合

- 固定データで先頭から最後まで実行できるか。
- 途中の確認表示と提出する最終報告を区別しているか。
- `pass` が残っていないか。

## テスト確認

| 種類 | 最低限の入力 | 確認すること |
| --- | --- | --- |
| 通常 | 有効な2〜4件 | FIFO順、開始、終了、待ち時間 |
| 同時到着 | 到着時刻が同じ2件 | 供給順、開始、終了、待ち時間 |
| 降順 | 到着2分の後に1分 | 全体無効と固定理由 |
| 境界 | 1枚、40枚、10件 | 許される最小、最大、件数上限 |
| 無効 | 0枚、41枚、到着時刻-1 | 除外と個別の理由 |
| 空 | `[]` | 空の結果、平均0、異常終了しないこと |

`assert` は中核で要求しません。期待値、実際値、一致したかの表示または表を使います。失敗時は期待値を実行後に書き換えず、入力、実際値、修正関数、再実行結果を残します。

## 形成的評価

- C9.O1: 各関数の責務が一文で説明され、供給された入力と戻り値に対応しているか。
- C9.O2: 通常、同時到着、降順、境界、その他の無効、空に、実行前の期待値と実行後の実際値があるか。
- C9.O3: R1〜R5の各判断から実行結果またはテストへたどれるか。

未確認事項を「満たした」とせず、「一部満たした」または「未確認」と記録できることも評価します。

## 支援の方法

- 完成行を示す前に、失敗した関数の引数、期待値、実際値を確認する。
- 関数シグネチャ、行の意味、空の出力リスト、理由文字列、`append` の一行例、比較用helper、チェックポイント表は供給するが、統合済み完成コードは配布しない。
- 入力が難しい場合は、自然言語設計やテスト表を口頭、カード、共有入力で作成できるようにする。
- ペア作業では設計者、トレーサー、テスト担当を交代し、最終説明は各自が行う。
- 個人の診断名や支援歴を成果物へ記録させない。

## 任意の発展

中核と四種類のテストが完成した学習者だけ、短い印刷優先へ進めます。必要なら並べ替えの機構は教師が供給します。学習者は自然言語規則、同数時の受付順、FIFOとの待ち時間比較、長い依頼が後回しになる限界、追加テストを担当します。発展の未実施を減点しません。

## 想定される誤解

- `pass` を残したスターターを完成品と考える。
- FIFOの中核を終える前に短い印刷優先へ進む。
- 関数内で `project_jobs` を直接使い、別入力で試せなくする。
- エラーが出なければ値も正しいと判断する。
- 無効入力のテストで理由まで確認しない。
- 同時到着をID順へ並べ替える、または降順入力を有効なFIFO入力として処理する。
- 空入力と無効な速度を同じ意味にする。
- 平均待ち時間だけで方式を評価し、順序の明確さやモデルの限界を書かない。
- 合成データの結果を実際の学校運用へ一般化する。

## Unit C累積チェックポイント

正式な5〜6時限の外で25分を確保します。固定入力は貸出依頼 `[[A,0,2],[B,0,1],[C,2,4]]`、一窓口、1冊/分、同時到着は入力順、冊数0以下は無効です。用紙には次の四枠をあらかじめ設けます。

1. **状態と構造（5分）**: 現在時刻と残件を変数・リスト行で示し、入力、主記憶上の状態、出力を分ける。
2. **契約と手順（6分）**: 検査、FIFO、時刻計算を三つの責務へ分け、有限な自然言語手順と停止条件を書く。
3. **トレースとテスト（6分）**: A、B、Cの開始は0、2、3、終了は2、3、7、待ちは0、2、1とする期待表を完成させ、空、`[D,3,0]`、同時到着を試す。
4. **情報構造・試作・限界（3分）**: B4の見出し順で結果カードを並べ、B6の低コスト試作として読取り順を確認し、「一窓口・固定速度」の仮定と限界を書く。

続く5分で、誤ってAとBを入れ替えた提供トレースを契約に照らして修正し、同時到着と空を再確認します。九つのC概念、B4、B6、期待値、修正、再確認をチェック欄で照合します。一項目でも未達なら該当するC1〜C8の自己確認へ戻してからC9成果物を再提出します。
