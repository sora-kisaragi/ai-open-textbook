# 教師用ガイド: インターネット、Web、情報サービス

## レッスンのねらい

一つの架空URLを分解し、URL入力、DNS、経路制御、伝送制御、TLS、HTTP、サービス内処理、画面表示を一続きの経路として追跡します。同じ「開かない」でも、観察証拠により疑う層が変わることを扱います。

## 前提確認

D1のクライアント、サーバー、スイッチ、ルーター、IPアドレス、プロトコル層を、LANからWANを越える経路として確認します。

## 学習者用本文との対応

1. インターネット、Web、情報サービスとURI・URLを区別する。（D2.O1）
2. `https://library.example.test/books/B204?view=summary` を構成要素へ分ける。（D2.O1）
3. URL入力から表示までの九段階を、層と機器を含めてたどる。（D2.O2）
4. 観察証拠からDNS、経路、伝送、TLS、HTTP、処理の候補を絞る。（D2.O3）
5. 性能、最新性、アクセシビリティ、安全性、運用のトレードオフを評価する。（D2.O3）

## 時間配分の目安

70〜110分です。前半で用語とURL分解、後半で経路カード、障害切り分け、サービス評価を扱います。

## 進め方と形成的評価

URLの四部分と九段階の経路カードを並べ替え、各段階で観察できる証拠を一つ添えさせます。
DNS問い合わせ自体もリンク、IP、トランスポートを使い、その後にWebサービスへの別の通信が始まるという入れ子を補足します。
D2.O1は分類理由、D2.O2はDNS・経路・伝送・TLS・HTTPを含む順序、D2.O3は観察と候補の整合性で確認します。

TLSの導入では「HTTPSだから全部安全」ではなく、通信路上の機密性・完全性とサーバー認証を支える範囲に限定します。証明書警告が出た場面では接続を止め、担当者へ報告するものとし、警告を無視して進む実習は行いません。

## 想定される誤解

- インターネットとWebを同義とする。
- DNSがURL全体や検索結果を処理するとする。
- ルーターがサーバー内のパスを読んで経路を選ぶとする。
- HTTPSがページ内容や運営者の信頼性を保証するとする。
- 失敗原因をすべてネットワークにする。

## 支援とアクセシビリティ

用語表、URL分解表、番号付き経路カードを用意します。図を使わず、段階、観察証拠、候補原因の三列で答える形式も認めます。URLは読み上げやすいよう、スキーム、ホスト名、パス、クエリを別行にも示します。

## 安全性と範囲

`.test` の名前と `192.0.2.24` のアドレスは教材用です。実在サイトへの接続確認、証明書警告の回避、ネットワーク診断コマンドの実行は不要です。教師が実演する場合も、学校の許可済み環境と公開済み情報だけを使います。

## 合成データと出典上の注意

RFC 2606は `.test`、RFC 3986はURI構成要素、RFC 9499はDNS用語、RFC 9293はTCPを確認する一次資料として限定利用します。
RFC 9110はHTTPと `http` / `https` URI、RFC 9846はTLS 1.3の保護範囲、RFC 9525はTLSでのサービス名照合にだけ使います。
W3C Web ArchitectureはWeb上の識別と相互作用の概念にだけ使います。
九段階の経路、障害表、サービス評価はプロジェクト独自です。

## 練習問題

- **インターネット、Web、URI・URL、サービスを区別する**（`prob.info1.networks.internet.web.001.v1`）: D2.O1
- **図書検索URLの経路をたどる**（`prob.info1.networks.internet.web.002.v1`）: D2.O2
- **観察から失敗した層を診断する**（`prob.info1.networks.internet.web.003.v1`）: D2.O3
- **サービスの性能と運用上の限界を評価する**（`prob.info1.networks.internet.web.004.v1`）: D2.O1〜O3
