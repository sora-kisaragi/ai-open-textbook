# アルゴリズムと効率

## 学習目標

1. アルゴリズムを、構造化した自然言語またはコードで表現できる。（C7.O1）
2. アルゴリズムを、正しさ、明確さ、入力が増えたときの仕事量で比較できる。（C7.O2）
3. テストとトレースを使ってアルゴリズムを改善できる。（C7.O3）

## 前提知識

- 繰り返し、リスト、添字を使えること。
- 関数の入力、戻り値、責務を説明できること。

## 印刷する順序を決める手順

星見高校メディアラボに複数の印刷依頼が届きました。**アルゴリズム**は、問題を解くための有限で明確な手順です。このレッスンの中核では、先に受け付けた依頼から処理する**受付順方式（FIFO）**を扱います。

受付順方式で正しい順序とは、入力された依頼の順序を変えないことです。依頼を飛ばしたり、同じ依頼を2回処理したりしてはいけません。

## 構造化した自然言語で表す

受付順の印刷スケジュールを、次のように書けます。

```text
経過時間を0とする
入力された依頼を先頭から一件ずつ取り出す
    依頼の必要枚数を経過時間に加える
    依頼番号と、その依頼の完了時刻を表示する
すべての依頼を処理したら終了する
```

判断基準を省略せず、開始、取り出す順序、更新、出力、終了を明記します。

## 手作業でトレースする

次の合成データを使います。1枚の処理を1単位時間とする単純なモデルです。

| 受付順 | ID | 必要枚数 |
| ---: | --- | ---: |
| 0 | `M301` | 7 |
| 1 | `M302` | 3 |
| 2 | `M303` | 5 |

| 処理するID | 処理前の経過時間 | 加える枚数 | 完了時刻 |
| --- | ---: | ---: | ---: |
| `M301` | 0 | 7 | 7 |
| `M302` | 7 | 3 | 10 |
| `M303` | 10 | 5 | 15 |

途中の値を表へ記録することを**トレース**といいます。最終結果だけでなく、どの依頼で値がずれたかを見つけられます。

## 習った構文で印刷スケジュールを表示する

各依頼は `[ID, 必要枚数]` というリスト行で表します。

```python
def print_fifo_schedule(jobs):
    elapsed = 0
    count = 0
    for job in jobs:
        job_id = job[0]
        sheets = job[1]
        elapsed = elapsed + sheets
        print(job_id, elapsed)
        count = count + 1
    if count == 0:
        print("依頼なし")


jobs = [
    ["M301", 7],
    ["M302", 3],
    ["M303", 5],
]

print_fifo_schedule(jobs)
```

実行結果は次のとおりです。

```text
M301 7
M302 10
M303 15
```

この補助関数は、入力されたリストを先頭からたどり、印刷順と完了時刻を表示します。リストを複製したり、並べ替えたりする高度な仕組みは必要ありません。

## 期待値と実際値を比べる

まず、実行前に期待する表示を書きます。

| テスト | 入力 | 期待する表示 |
| --- | --- | --- |
| 通常 | `M301:7`, `M302:3`, `M303:5` | `M301 7`、`M302 10`、`M303 15` |
| 1件 | `M310:4` | `M310 4` |
| 空 | 依頼なし | `依頼なし` |

実行後は、表示されたIDの順序と完了時刻を一行ずつ照合します。単にエラーが出なかったことを成功とはしません。

## 入力が増えたときの仕事量

受付順方式では、各依頼を一度ずつ読み、表示します。依頼が3件なら3回、6件なら6回処理します。入力が2倍になると、中心となる繰り返しの回数もおおむね2倍になります。

コードの行数だけで効率を判断しません。「入力が増えたとき、どの確認や比較が何回増えるか」を考えます。

## 任意の発展: 短い印刷を優先する

必要枚数が最も少ない依頼から処理する方式も考えられます。ただし、これは中核課題ではありません。取り組む場合は、教師から並べ替えの仕組みを供給してもらうか、次の自然言語手順をカードでたどります。

1. 未処理の依頼から必要枚数が最も少ないものを探す。
2. 同じ枚数なら、受付が早いものを選ぶ。
3. 選んだ依頼を処理済みにする。
4. 未処理がなくなるまで繰り返す。

同じ依頼データを使った場合、短い依頼を先にすると一部の待ち時間が小さくなることがあります。一方で、長い依頼は受付順方式より後になることがあります。入力順を保つことと、この固定データでの待ち時間のどちらを要件に含めるかを区別して判断します。

## よくある間違い

- 手順に曖昧な語を残し、人によって異なる結果になる。
- 最終結果だけを見て、途中の経過時間をトレースしない。
- 空の入力や1件だけの入力を試さない。
- コードが短いほど、入力が増えたときも必ず効率がよいと考える。
- 任意発展の短い印刷優先を、全員にとって唯一の正解だと考える。

## まとめ

- アルゴリズムは、明確で有限な問題解決手順です。
- 中核の受付順方式では、入力順を保って各依頼を一度ずつ処理します。
- トレースと期待値・実際値の照合で、順序と更新を確かめます。
- 効率は、入力が増えたときの仕事量で比較します。
- 短い印刷優先は、追加の規則とテストを伴う任意の発展です。
