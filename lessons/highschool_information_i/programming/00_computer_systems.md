# コンピュータのしくみと情報処理

## 学習目標

1. 入力、主記憶、保存、処理、出力と、OS・アプリケーションの役割を関係付けて説明できる。（C1.O1）
2. アプリケーションが扱う値と状態の変化を手作業で追い、OSによる資源利用の仲介と分けて説明できる。（C1.O2）
3. 表現の範囲と、主記憶・保存容量・時間・装置などの資源の限界を区別できる。（C1.O3）

## このレッスンの見通し

**大きな問い:** 画面で「印刷」を押してから紙が出るまで、情報はどこにあり、何によって変化するのでしょうか。

- 時間: 50分×2時限
- 用意するもの: 筆記用具、役割カード用の小さな紙7枚（紙がなければノートの表でよい）
- 進め方: 第1時は役割と状態の追跡、第2時は誤りの修正と別場面への応用を行います。

この教材で使う学校名、依頼番号、ファイル名、数値はすべて架空です。実在する生徒や機器の記録ではありません。

## 準備チェック

次の三つを、短い言葉で答えてください。

1. キーボードで文字を入れる操作は、入力と出力のどちらですか。
2. 画像を細かく表すほど、一般にデータ量は増えますか、減りますか。
3. 「保存したファイル」と「いま画面で編集中の内容」は、いつでも同じだと思いますか。

1と2に答えられ、3で「同じとは限らない」と説明できたら、そのまま進みます。
1か2で迷ったら、入力はコンピュータへ情報を渡すこと、出力は結果を人や別の装置へ渡すことを確認してから進みます。
3で「いつも同じ」と答えたら、「保存後に一文字追加し、まだ保存していない場面」を想像してから、もう一度答えます。
それでも二つ以上説明できないときは、先生や学習支援者と例を一つ作ってから先へ進みます。

## コンピュータを役割で捉える

機種やプログラミング言語が変わっても、情報処理を次の役割に分けると流れを説明できます。

| 役割 | 何をするか | 印刷での例 |
| --- | --- | --- |
| 入力 | 外からデータや指示を受け取る | 部数を選び、印刷を指示する |
| 保存装置 | あとで再び使うデータを保持する | 画像ファイルを保存する |
| 主記憶（メモリ） | 実行中に使う命令やデータを一時的に置く | 読み込んだ画像や印刷中の残り枚数を置く |
| 処理装置 | 命令に従って計算や判断を行う | 必要な枚数を計算する |
| 出力 | 結果を人や別の装置へ渡す | 状態を画面に示し、紙へ印刷する |
| ソフトウェア | 処理装置が実行する命令や、処理に使うデータをまとめる | 作図アプリケーションとOS |

**アプリケーション**は利用者の目的に近い処理を受け持ちます。

**OS**（オペレーティングシステム）はソフトウェアの一つで、処理装置を使う時間、主記憶の領域、ファイル、入力装置、出力装置などをアプリケーションが利用できるよう仲介します。
OSは処理装置や主記憶そのものではなく、アプリケーション固有の計算内容を代わりに決めるものでもありません。

ここでは、二つの層を分けて考えます。

1. **アプリケーションの状態・流れの層**: 入力値を受け取り、保存済みデータを必要に応じて読み、主記憶上の実行中の状態を処理によって変え、結果を出力する。
2. **資源利用の仲介の層**: OSが、主記憶の領域、処理装置を使う時間、ファイル、入力装置、出力装置を利用するための共通の働きを提供する。

![上段に入力からアプリケーションの状態、処理、出力までの値の流れ、下段に保存装置を置き、OSが主記憶・処理装置・保存装置・入出力装置の利用を仲介することを破線で重ねた二層図](figure:c1-computer-system-trace.svg)

### 図のテキスト等価版

図を使わない場合は、次の表が同じ内容を表します。左列の状態変化を先に追い、右列のOSの働きはその上へ重ねます。

| 段階 | アプリケーションの値と状態の流れ | OSによる資源利用の仲介 |
| ---: | --- | --- |
| 1 | 利用者からファイル名、ページ数、部数、印刷指示を受け取る | 入力装置から情報を受け取るための共通の働きを提供する |
| 2 | 保存済みファイルを読み、必要なデータと印刷依頼の状態を主記憶へ置く | ファイルと主記憶領域の利用を仲介する |
| 3 | 処理装置がアプリケーションの命令を実行し、必要枚数を計算して状態を更新する | 処理装置を使う時間と主記憶領域の利用を仲介する |
| 4 | アプリケーションが画面表示と印刷用データを出力先へ渡す | 画面とプリンターの利用を仲介する |
| 5 | 完了した枚数やエラーの有無を状態として残す | 必要に応じて、装置から届く完了や失敗の情報をアプリケーションへ渡す |

実際のコンピュータでは処理が重なり、OS、アプリケーション、装置の分担も実装によって異なります。この図と表は、一つの依頼について「値がどう変わるか」と「資源利用を誰が仲介するか」を混同しないための概念モデルです。

## 保存装置と主記憶は、なぜ分けるのか

どちらもデータを保持しますが、同じ役割ではありません。

| 比べる点 | 保存装置 | 主記憶 |
| --- | --- | --- |
| 主な役割 | 後で使うデータを保つ | いま実行している処理のデータを置く |
| 印刷依頼で置くもの | 保存済みの画像ファイル | 読み込んだ画像の必要部分と処理中の状態 |
| 印刷での例 | `festival_map.pdf` | 開いた複数ページ文書の必要部分、部数、処理状態 |
| 容量が足りないとき | 新しいファイルや待ちデータを保存しにくい | 大きなデータを一度に処理しにくい |

たとえば、保存済みの画像を開いて文字を追加する場面を考えます。
保存装置には読み込み元のファイルがあり、主記憶には編集中の状態を置きます。
編集した状態を保存装置上のファイルへ反映する処理と、主記憶上で編集を続ける処理は分けて追います。

このレッスンでは、主記憶を**揮発性メモリ**として扱います。揮発性メモリは、電源を切ったり、停電などで電力が失われたりすると、保持していた内容が失われるメモリです。
そのため、このモデルでは、主記憶だけにある未保存の編集内容や処理途中の状態は、電力が失われると残りません。
ただし、すべてのファイルやコンピュータ内のすべての状態が失われるという意味ではありません。保存装置へ書き込まれた内容は、主記憶の一時状態とは分けて考えます。

このレッスンの概念モデルでは、大きなファイルを保存装置から必要な部分だけ主記憶へ読み込む処理も考えます。
「保存できる大きさ」と「一度に主記憶へ置く大きさ」は別の条件です。
実際の方法はOS、アプリケーション、データ形式によって異なります。

## 状態とトレース

**状態**とは、ある時点で処理に関係する値の組です。印刷依頼なら、`受付済み`、`処理中`、`完了`という進み具合、残り枚数、エラーの有無などが状態です。

**トレース**とは、入力や処理が起きるたびに状態がどう変わったかを、順番に記録することです。結果だけでなく途中を残すので、予想と違う出力が出た場所を探せます。

## 考え方を追う例: 印刷依頼Q104

架空の印刷依頼を、コードを使わず手で追います。両面印刷は扱わず、1ページを紙1枚に印刷する単純なモデルです。

| 項目 | 値 |
| --- | --- |
| 依頼番号 | `Q104` |
| 保存済みファイル | `festival_map.pdf`（6ページの文書） |
| 部数 | `2` |
| 印刷方法 | モノクロ |

まず、必要な枚数は `6ページ × 2部 = 12枚` です。ページ数と部数は入力、掛け算は処理、その結果である12は次の処理に使う状態になります。

次の表は、各出来事の**直後**の状態を記録した手作業トレースです。行を一つずつ読み、次の行を隠して予想してから確かめてください。

| 段階 | 起きたこと | 保存装置 | 主記憶にある主な状態 | 処理・出力 |
| ---: | --- | --- | --- | --- |
| 0 | 印刷前 | 6ページの画像がある | 印刷依頼なし | 出力なし |
| 1 | 利用者が部数2で印刷を指示 | 画像は変わらない | `状態=受付済み`、`部数=2` | 画面に受付を表示 |
| 2 | アプリケーションが画像を読み込み、依頼を準備する | 画像は残る | `状態=準備中`、画像の必要部分 | OSがファイル、主記憶領域、処理装置の利用を仲介 |
| 3 | 枚数を計算する | 画像は変わらない | `状態=処理中`、`残り=12` | `6×2=12` と計算 |
| 4 | 1枚を出力する | 画像は変わらない | `状態=処理中`、`残り=11` | プリンターから1枚出る |
| 5 | 残りも順に出力する | 必要なら処理記録を保存 | `状態=完了`、`残り=0` | 合計12枚、画面に完了を表示 |

ここで大切なのは、保存済み画像が毎段階で書き換わるわけではないことです。
変化している中心は、アプリケーションが主記憶上で扱う依頼の状態と残り枚数です。
枚数の意味と計算手順はアプリケーションの命令にあり、処理装置がその命令を実行します。
OSの役割は、その実行に必要な資源と装置の利用を仲介することです。

## 対比: 説明になっていない例

> 印刷ボタンを押すとCPUがファイルを永久に覚え、OSが12枚を計算して、すぐプリンターが動く。

この説明には三つの混同があります。

| 混同 | 修正 |
| --- | --- |
| 処理装置がファイルを永久に保存する | 保存済みファイルは保存装置、実行中のデータは主記憶と分ける |
| OSだけが計算する | OSは資源利用を仲介し、処理装置がソフトウェアの命令を実行する |
| 指示の直後に出力が完了する | 受付、読み込み、計算、装置への送信、出力という状態変化をたどる |

## 表現と資源の限界

コンピュータは有限のビット数と資源で処理します。学習用に、ページ数を符号なし8ビットで表す欄を考えます。1ビット増えると組合せは2倍になります。

| ビット数 | 区別できる値の個数 | 0から始めた最大値 |
| ---: | ---: | ---: |
| 1 | 2 | 1 |
| 2 | 4 | 3 |
| 3 | 8 | 7 |
| 8 | 256 | 255 |

0も一つの値なので、8ビットでは0から255までです。この架空の欄へ260ページはそのまま入りません。これは**表現の範囲**の問題です。実在する印刷システムが8ビットでページ数を保存するという意味ではありません。

一方、画像を表現できても、空いている主記憶が12 MBで、1ページの展開に4 MB必要なら、単純なモデルでは同時に置けるのは `12÷4=3` ページです。4ページ以上は分割して処理できる場合があります。これは**資源**と処理方法の問題です。

温度23.47度を0.1度刻みで保存して23.5度にする場合は、範囲外ではなく**丸め**です。差は `23.5-23.47=0.03` 度です。

## やってみよう

役割の配置、誤解の修正、途中状態、限界の分類を順に行います。各活動では、アプリケーションの状態変化とOSによる資源利用の仲介を同じ説明へ混ぜないでください。

### 1. 役割カードを並べる

「マイクで声を録音し、ファイルとして保存してから再生する」場面について、まず入力、アプリケーションの状態、主記憶、処理、保存、出力の値の流れを並べます。
次に、OSカードを流れの途中の一段階にはせず、主記憶、処理装置、保存装置、入出力装置の利用を仲介する位置へ置きます。
各カードに、保持または伝達されるデータを一つ書きます。

迷ったら、最初に「声」を入力へ、「音」を出力へ置きます。その間で、いま処理する音のデータは主記憶、あとで再生するファイルは保存装置と考えます。

### 2. 誤解を直す

次の文の誤りを二つ見つけ、役割を示す言葉を使って書き直してください。

> 保存ボタンを押す前でも、編集中の内容は必ず保存装置にあり、処理装置が電源を切った後も覚えている。

「必ず」と「処理装置が覚えている」に注目します。書き直しには、主記憶と保存装置の両方を使います。

### 3. 途中状態を埋める

3ページを2部印刷します。段階3の直後を `状態=処理中、残り=6` とします。2枚出力した直後の状態と、すべて出力した直後の状態を書いてください。答えには状態名と残り枚数の両方を含めます。

### 4. 限界を分類する

次を「表現範囲」「丸め」「主記憶などの資源」に分け、理由を一文で説明します。

- 0〜255の欄へ300を入れようとした。
- 18.26を0.1刻みで18.3と記録した。
- 保存済み動画はあるが、一度に開くための主記憶が足りない。

## 別の場面への転移

印刷とは異なる装置場面へ、値と状態の流れ、OSによる資源仲介、表現と資源の限界を自力で移します。

### 飲料の自動販売機

印刷とは異なる場面で確かめます。ここでは、OSを備え、その上で販売アプリケーションが動く学習用の自動販売機を考えます。
硬貨を入れ、商品ボタンを押すと、在庫があれば商品とおつりが出ます。
実機には制御ソフトウェアが装置を直接扱う構成もありますが、それはOSがアプリケーションの資源利用を仲介する今回のモデルとは分けます。

1. 入力、保存装置、主記憶上のアプリケーション状態、処理、出力の流れを表にし、OSによる資源利用の仲介を別欄へ示す。
2. 初期状態を `投入金額=0、選択=なし、在庫=2` とし、「150円を入れる」「120円の商品を選ぶ」「商品を1個出す」の各直後をトレースする。
3. このレッスンと同じく、主記憶を揮発性メモリとして扱う学習用モデルで考える。途中で電源が切れたとき、主記憶から失われてもよい一時状態と、失うと困るため電源が切れる前に保存装置へ書き込むべき状態を一つずつ挙げ、理由を書く。
4. 投入金額を0〜255の欄へ記録する学習用モデルで、300円を記録しようとした場合を考え、状態の変化と表現範囲の限界を分けて説明する。

20分の独立転移では、A4判1枚のワークシートにまとめます。
上半分を「出来事｜直後のアプリケーション状態｜入出力」、別の右欄を「OSが仲介する資源」とします。
下半分に、一時状態・保存すべき状態・表現範囲の3行を置いてください。
5分の確認では、出来事ごとの状態が3段階以上あり、OS欄が状態欄と混ざっていないかを先に照合します。

成功の目安は、次の四点です。

- 各役割が受け取るものと渡すものを区別する。
- どの出来事でどの状態が変わったかを示す。
- 一時状態と保存すべき状態を理由付きで分ける。
- 300円の事例を装置故障ではなく、学習用の値の範囲に関する限界として説明する。

## 到達チェック

次の四項目を、本文を閉じて確認します。

1. 主記憶と保存装置の違いを、電源または編集中データの例で説明できる。
2. OSを装置とせず、アプリケーションと装置利用の仲介として説明できる。
3. 一つの入力について、各出来事の直後の状態を3段階以上トレースできる。
4. 表現範囲、丸め、資源不足を別の原因として分類できる。

4項目すべてに根拠付きで答えられたら、練習へ進みます。
2〜3項目なら、誤った項目に対応する「保存装置と主記憶」「状態とトレース」「表現と資源の限界」を読み直し、`やってみよう` の同じ番号をもう一度行います。
0〜1項目、または読み直しても主記憶・保存装置・処理装置の区別ができない場合は、先生や学習支援者と、Q104の段階0、1、3、4だけを使った短縮トレースを作ります。
退出時には、入力値6と2、計算結果12、1枚出力後の残り11を記録し、OSが仲介する資源を一つ別欄へ書けばよいものとします。
完全なQ104表、限界の三分類、自動販売機の転移は自己学習で再試行してください。

## よくある間違い

- 処理装置と主記憶を同じものと考える。処理装置は命令を実行し、主記憶は実行中の命令やデータを置きます。
- 主記憶と保存装置を速度だけの違いと考える。いま使う状態と、後で使う保存済みデータという役割も異なります。
- OSをアプリケーションや装置そのものと考える。OSは装置や資源の利用を仲介するソフトウェアです。
- 出力結果だけを書き、途中状態を残さない。出来事の直後ごとに値を記録すると、変化した場所を探せます。
- 「デジタルなら正確で無限」と考える。値の範囲、刻み、容量、時間はいずれも有限です。
- 表現範囲と資源不足を同じ原因にする。値を形式に収められるかと、収めたデータを処理できるかを分けます。
- 8ビットの例を実在製品の仕様と考える。限界を考えるための架空のモデルです。

## まとめ

- コンピュータは、入力、保存、主記憶、処理、出力と、それらを動かすソフトウェアの関係で捉えられます。
- OSは、アプリケーションと処理装置・主記憶・入出力装置・ファイルの利用を仲介します。
- 状態はある時点の値の組で、トレースは出来事ごとの状態変化の記録です。
- 主記憶と保存装置を分けると、編集中の状態、保存済みデータ、容量不足の違いを説明できます。
- 表現の範囲、丸め、資源の限界は、原因を分けて考えます。

## 任意発展: 学校図書館の貸出端末

学校図書館の貸出端末で、利用者カードと本を読み取り、貸出結果を表示するまでを考えます。入力、保存装置、主記憶、処理、出力、OS・アプリケーションへ分け、貸出前、読取後、処理後の状態を表にしてください。さらに、結果は表せるが処理できない資源上の限界を一つ挙げます。
