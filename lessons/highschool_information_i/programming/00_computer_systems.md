# コンピュータのしくみと情報処理

## 学習目標

1. オペレーティングシステムが、ソフトウェアと処理装置・主記憶・入出力装置の間を仲介する役割を説明できる。（C1.O1）
2. 一つの命令とデータがコンピュータの中を進む流れをたどれる。（C1.O2）
3. デジタル表現と資源が有限であることを、計算の限界と結び付けて説明できる。（C1.O3）

## 前提知識

- B2で学んだ、文字、画像、音が一定の規則で数値として表現されること。
- B2で学んだ、表現の細かさとデータ量のトレードオフ。

## 導入: 印刷ボタンの先で何が起こるか

架空の星見高校メディアラボでは、授業で作ったポスターを共有プリンターで印刷します。利用者が「印刷」を選んでも、紙がすぐに出るとは限りません。印刷依頼はデータとして受け取られ、印刷待ち行列に保存され、順番に処理されます。

この教材で使う依頼番号、ファイル名、ページ数はすべて架空の合成データです。実在する生徒の氏名や作品は含みません。

## コンピュータを役割で捉える

コンピュータの機種が変わっても、情報処理は次の役割に分けて考えられます。

| 役割 | 印刷待ち行列での例 |
| --- | --- |
| 入力 | 利用者がページ数や印刷方法を選び、印刷依頼を送る |
| 記憶 | 依頼データや処理中の画像を一時的に保持する |
| 処理 | 印刷順を決め、ページ画像をプリンター用データへ変換する |
| 出力 | 受付結果を画面に表示し、プリンターが紙へ印刷する |
| ソフトウェア | 入力の解釈、待ち行列の管理、装置への命令を行う |
| 外部装置 | キーボード、画面、保存装置、プリンターなど |

処理装置は命令を実行します。この教材では、実行中の命令やデータを置く主記憶と、あとで取り出すデータを保持する保存装置を、情報の流れを考えるための役割として区別します。実際の機器では役割が複数の部品に分かれたり、一つの部品が複数の役割を持ったりします。

アプリケーションが、処理装置やプリンターを毎回直接制御するわけではありません。

**オペレーティングシステム（OS）**は、アプリケーションからの依頼を受け、処理装置を使う時間、主記憶の領域、入力装置から届くデータ、出力装置へ送る命令を調整します。印刷の例では、アプリケーションが印刷を依頼し、OSが依頼データを受け取って待ち行列や装置へ仲介します。

したがって、「ソフトウェアが印刷する」を詳しくすると、アプリケーション、OS、処理装置、主記憶、出力装置の役割へ分けられます。OSもソフトウェアであり、処理装置や主記憶そのものではありません。

## データと命令の流れをたどる

架空の印刷依頼 `Q104` を考えます。

| 項目 | 値 |
| --- | --- |
| 依頼番号 | `Q104` |
| ファイル名 | `festival_map.png` |
| ページ数 | `6` |
| 部数 | `2` |
| 印刷方法 | `mono` |

流れは次のようにたどれます。

1. **入力**: ソフトウェアが、ページ数 `6`、部数 `2` などを受け取る。
2. **記憶**: 依頼データを主記憶へ置き、待ち行列の末尾へ加える。
3. **処理**: `6ページ × 2部 = 12枚` と計算し、印刷可能かを確かめる。
4. **装置への命令**: アプリケーションの依頼をOSが仲介し、プリンターへページデータと印刷設定を送る。
5. **出力**: 画面に状態を表示し、プリンターが12枚を出力する。

この教材では、`ページ数 × 部数` で求めた紙の数を**枚数**と呼びます。両面印刷は扱わず、1ページを紙1枚に印刷する単純なモデルとします。

「コンピュータが印刷した」で終わらず、どのデータが、どの役割を通り、どの出力になったかを順に説明します。

## デジタル表現には範囲がある

コンピュータは、無限に細かい値や無限に大きい数を、そのまま保存できません。決められたビット数で表す場合、表現できる値の個数には限りがあります。

学習用の単純な例として、ページ数を符号なし8ビットで保存するとします。1ビット増えるたびに組合せは2倍になるので、次のように数えられます。

| ビット数 | 区別できる値の個数 | 0から始めた最大値 |
| ---: | ---: | ---: |
| 1 | 2 | 1 |
| 2 | 2 × 2 = 4 | 3 |
| 3 | 2 × 2 × 2 = 8 | 7 |
| 8 | 2を8回掛けた256 | 255 |

計算手順を、特定のプログラミング言語を使わない擬似コードで表すと次のようになります。

```text
値の個数を1とする
8回繰り返す
    値の個数を2倍にする
最大値を「値の個数 - 1」とする
```

8ビットでは256個の値を区別できます。0も一つの値として数えるため、表せる整数は0から255までです。

この学習用設計では、ページ数 `260` はその欄の表現範囲を超えます。実際のシステムがページ数を8ビットで保存しているという意味ではありません。どの表現規則を選ぶかによって、扱える範囲が決まることを示すモデルです。

範囲内でも、細かい値を有限の刻みへ丸めると**丸め誤差**が生じます。たとえば、温度 `23.47` 度を小数第1位まで保存する規則なら、保存値は `23.5` 度です。

差は `23.5 - 23.47 = 0.03` 度で、元の `23.47` を保存値だけから完全には復元できません。

- **範囲の限界**: `260` を0〜255の形式へそのまま収められない。
- **精度の限界**: `23.47` を0.1度刻みへ丸めると `23.5` になり、差が残る。

どちらも有限な表現による限界ですが、「大きさが範囲外」と「範囲内の値を丸めた」は別の問題です。

## コンピュータの資源にも限りがある

値を表現できても、必ず処理できるとは限りません。

- 主記憶が足りなければ、大きな画像を同時に展開できないことがある。
- 保存容量が足りなければ、新しい印刷データを待ち行列へ保存できないことがある。
- 処理装置や通信の速度が有限なので、依頼が増えると待ち時間が長くなる。
- プリンターの用紙、インク、1分あたりの印刷枚数にも上限がある。

たとえば、処理用の空き主記憶が12 MBで、ページ画像1枚の展開に4 MB必要なら、単純化したモデルでは同時に3枚までしか置けません。ページ数が4枚以上でも印刷を分割すれば処理できるかもしれませんが、全ページを一度に置く設計では失敗します。

限界を考えるときは、次の二つを分けます。

1. **表現の限界**: その値を決められた形式で表せるか。
2. **資源の限界**: 表せるデータを、利用できる記憶容量、時間、装置で処理できるか。

## よくある間違い

- 処理装置と主記憶を同じものと考える。処理装置は命令を実行し、主記憶は実行中の命令やデータを置きます。
- ソフトウェアを物理的な部品だと考える。ソフトウェアは、処理装置が実行する命令やデータの集まりです。
- OSをアプリケーションや処理装置そのものと考える。OSは、アプリケーションと処理装置・主記憶・入出力装置の利用を仲介するソフトウェアです。
- 「デジタルなら正確で無限」と考える。表現の細かさ、値の範囲、容量、時間はいずれも有限です。
- 範囲外と丸め誤差を同じ原因として扱う。表現できない大きさか、刻みへ丸めた差かを区別します。
- 容量不足と値の範囲外を同じ原因として扱う。どの表現または資源が限界になったかを特定します。
- この教材の8ビットモデルを、実在する印刷システムの仕様だと考える。これは限界を調べるための架空の単純化です。

## 自己確認の診断と修正

確認できた状態は、部品名を挙げるだけでなく、一つのデータを役割に沿って追跡し、表現の限界と資源の限界を分けて説明できることです。

- 流れが止まる場合は、「何を受け取るか」「どこへ置くか」「何を計算するか」「何を出すか」の順に書きます。
- `256` 個の値と最大値 `255` が混同した場合は、`0` も一つの値として数えます。
- 「性能が足りない」だけの場合は、記憶容量、処理時間、通信速度、装置能力のどれかを特定します。

## まとめ

- コンピュータの情報処理は、入力、記憶、処理、出力と、それらを制御するソフトウェアの役割で捉えられます。
- 命令とデータの流れをたどると、どこで何が起きたかを説明できます。
- デジタル表現の範囲と、記憶容量・時間・装置などの資源は有限であり、計算できることに限界を与えます。
- OSはアプリケーションと処理装置・主記憶・入出力装置の利用を仲介します。
- 有限な数値表現では、範囲外だけでなく丸めによる差も生じます。
