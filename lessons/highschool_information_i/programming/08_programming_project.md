## 準備チェックと復習

C1〜C8から、状態表、条件の真偽、停止条件、リスト行、関数契約、探索手順、モデルの仮定を各一つ説明します。七項目すべてが開始条件であり、特にC6の関数契約、C7の有限な手順とテスト、C8の仮定と限界は省略できません。一項目でも未達なら該当節の自己確認へ戻り、最小入力のカードまたは表で再現してから中核課題へ進みます。迷ったら失敗した一項目だけを直し、説明できてから七項目を再確認します。

## 考え方のモデル

プロジェクトは「要件→言語独立の契約と状態表→テスト契約→Python実装→期待値と実際値→修正→再テスト→限界」の循環で進めます。Pythonコードだけを成果にせず、実装前の概念成果物と実行証拠を対応付けます。

## 例題: FIFO契約を手で確かめる

到着時刻 `[0:A, 0:B, 2:C]` では、同時到着の入力順を保ちA、B、Cの順です。降順 `[2:C, 0:A]` は契約違反として受け付けません。「並べ替えて直す」は中核契約を変えるため行いません。

## 一緒に試す

二件の依頼について、要件R1〜R5、関数の責務、通常・同時到着・降順・空の期待結果を表へ埋めます。Pythonを開く前に、各テストがどの要件を確かめるかを線で結びます。

## パフォーマンス課題: 昼休み印刷キュー報告ツール

星見高校メディアラボは、昼休みに受け付けた印刷依頼について、受付順の処理と予測待ち時間を確認できる小さなツールを必要としています。依頼と数値はすべて架空の合成データです。実在する生徒名、利用履歴、文書内容は使いません。

中核課題は5〜6時限で完成できる範囲に限定します。第5時限までに中核実装と主要テストを終え、第6時限は修正、要件別評価、改善の再テストに使えます。

### 中核の範囲

- 一台のプリンターで、最大10件の依頼を扱う。
- 各依頼は `[id, arrival, sheets]` という固定したリスト行で表す。
- `arrival` は0以上の整数分、`sheets` は1〜40枚とする。
- 入力の到着時刻は小さくならない順（非減少順）とする。同じ到着時刻の依頼は、供給された順序を保つ。
- 処理方式は受付順（FIFO）とする。途中で到着時刻が小さくなる入力列は、FIFOの入力契約に反するため全体を無効とする。
- 印刷速度は1分4枚で固定し、処理時間は分単位で切り上げる。
- 出力は各依頼のID、開始時刻、終了時刻、待ち時間、平均待ち時間を含む。
- 無効な依頼は印刷順へ入れず、IDと理由を報告する。
- 空の入力では空の結果を返す。

必要枚数は「枚」で扱います。ページ数と部数から求める場合は `ページ数 × 部数 = 枚数` とします。両面印刷はこのモデルの対象外です。GUI、ファイル保存、ネットワーク、実機制御、認証も範囲外です。

## 1. 要件を検査可能な形にする

| 要件 | 確認方法 |
| --- | --- |
| R1 非減少の到着時刻を持つ有効な依頼を受付順のまま処理する | 通常順、同時到着、降順入力の期待値と実際値を比較する |
| R2 開始、終了、待ち時間を計算する | 手計算した小さな例と比較する |
| R3 無効な依頼を除外し、理由を示す | 0枚、41枚、負の到着時刻を入力する |
| R4 平均待ち時間を求める | 既知の待ち時間の合計を件数で割る |
| R5 空の入力でも異常終了せず、空の結果を返す | 空のリストで実行し、返り値を確認する |

「使いやすい」「速い」だけでは確認できません。入力、期待する動作、観察する結果を対応させます。

## 2. Python固有の部品を確認する

ここまでのレッスンでは、リストから値を読む方法を中心に扱いました。
このプロジェクトでは、計算した結果行を空のリストへ一件ずつ追加するために、Pythonの `append` を使います。
これはFIFOの考え方そのものではなく、結果をリストに組み立てるためのPython固有の操作です。

```python
results = []
result_row = ["M511", 0, 3, 0]
results.append(result_row)
print(results)
```

表示は `[["M511", 0, 3, 0]]` です。`append` は受け取った一要素を末尾へ加えます。この中核課題では、条件を一行へまとめる `or` は使わず、到着時刻と枚数を別々の `if` で検査します。

## 4. 最小実行可能なチェックポイント

一度に全部を書かず、次の順に進めます。

### チェックポイント1: 一件を検査する

- `validation_error(0, 1)` の期待値は空文字列。
- `validation_error(0, 0)` の期待値は枚数の理由。
- `validation_error(-1, 5)` の期待値は到着時刻の理由。

期待値と実際値を表示し、三つが一致してから次へ進みます。

### チェックポイント2: 有効な依頼だけを受付順で残す

- `M514` は除外する。
- `M511`、`M512`、`M513`、`M515` の順序は変えない。
- 同時到着の `[T2, 2, 8]`、`[T1, 2, 4]` は、`T2`、`T1` の供給順を保つ。
- 到着2分の後に1分が現れる降順入力は、全体を `到着時刻は非減少順` として無効にする。
- 無効な行のIDと理由を確認できるようにする。

### チェックポイント3: 時刻を一件ずつ計算する

- 最初は有効な2件だけで、開始、終了、待ち時間を手計算と比べる。
- 一件処理するたびに、プリンターが空く時刻を更新する。
- 各結果行の形を先に決める。

### チェックポイント4: 平均と空入力を扱う

- 待ち時間の合計と件数を繰り返しで求める。
- 件数が0のときに割り算をせず、0を返す。
- `simulate_fifo([], 4)` が異常終了せず、空の結果を返すことを確認する。

### チェックポイント5: 全体を接続する

固定データを検査、受付順、時刻計算、平均の順に渡します。途中の確認表示と最終報告を区別し、先頭から最後まで再実行します。

## 5. 通常・境界・無効・空をテストする

期待値は実行前に書きます。初心者向けの中核では `assert` を使わず、期待値、実際値、一致したかを表示するか、次の表へ記録します。

| 種類 | 入力例 | 確かめること | 期待値 |
| --- | --- | --- | --- |
| 通常 | 有効な2〜4件 | FIFO順、開始、終了、待ち時間 | 手計算して記入 |
| 同時到着 | 同じ到着時刻の2件 | 供給順を保つ | ID順と各待ち時間を記入 |
| 境界 | 1枚、40枚、10件 | 許される最小・最大・件数上限 | すべて有効 |
| 無効 | 0枚、41枚、到着時刻-1、降順入力 | 除外と理由 | 各理由を記入 |
| 空 | `[]` | R5と平均の扱い | 空の結果、平均0 |

供給された `show_check` を使うと、確認を同じ形で表示できます。関数の完成内容は自分で実装します。

```python
def show_check(label, expected, actual):
    print(label, "期待値:", expected, "実際値:", actual, "一致:", expected == actual)


expected = "ここへ実行前の期待値を書く"
actual = "ここへ関数の実行結果を入れる"
show_check("テスト名", expected, actual)
```

テストが失敗したら、入力、期待値、実際値、変更した関数、再実行結果を記録します。期待値が要件と矛盾していないかも確認します。

## 6. 要件に照らして評価する

R1〜R5を次の三段階で評価します。

- **満たした**: 対応する実行結果とテストを示せる。
- **一部満たした**: 動く条件と動かない条件を区別できる。
- **未確認**: 必要なテストまたは証拠がない。

「動いた」だけで終えず、たとえば「R1は通常順と同時到着で供給順を保ち、降順入力を理由付きで無効にした」「R3は0枚と41枚を除外した」のように証拠を書きます。有効なFIFO入力については、すべての結果行で `待ち時間 >= 0` であることも表示します。

## 7. 限界と改善を説明する

- 合成データと固定速度を使うモデルであり、現実の待ち時間を保証しない。
- 受付順は入力された順序を保つ。この固定データでは、別の順序にすると平均待ち時間が変わる場合がある。
- 最大10件、一台、整数分、片面印刷という範囲に限られる。
- 次に一つだけ改善するなら何か、その改善をどのテストで確かめるか。

### 任意の発展: 短い印刷優先

中核を完成して時間がある場合だけ、必要枚数が少ない依頼を先にする方式を検討できます。並べ替えの仕組みは教師が供給しても構いません。同数時の受付順、長い依頼が後回しになる問題、追加テストを明記します。発展を行わなくても中核課題の評価には影響しません。

## 5〜6時限の進行例

1. 第1時: R1〜R5、スターター、検査関数。
2. 第2時: 有効な依頼の抽出とFIFO順の確認。
3. 第3時: 開始、終了、待ち時間の計算。
4. 第4時: 平均、通常・境界・無効・空のテスト。
5. 第5時: 通常・境界・無効・空・件数上限のテストと修正。
6. 第6時: 要件別評価、限界、改善、再テスト、提出物点検。

5時限の場合は、要件別評価を各時限の最後へ分散し、第6時の改善を一つの小さな再テストへ絞ります。短い印刷優先は中核時間へ入れません。

## よくある間違い

- `pass` を残したまま、全体が完成したと報告する。
- FIFOの中核を終える前に任意発展へ進む。
- 関数内で固定データを直接読み、別の入力で試せなくする。
- 通常例だけを試し、0件、上限、無効入力を確認しない。
- 同時到着をID順へ並べ替える、または降順入力を有効なFIFO入力として処理する。
- エラーが出ないことだけを成功とし、期待値と実際値を比べない。
- 平均待ち時間だけで方式を評価し、順番の明確さや限界を書かない。
