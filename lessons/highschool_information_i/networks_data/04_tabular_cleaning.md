# 表形式データとクリーニング

## 学習目標

1. 表や分かち書き済みテキストの観測、変数、値、データ型、欠測を識別できる。（D5.O1）
2. 意味を黙って変えない、再現可能なクリーニング規則を適用できる。（D5.O2）
3. 変換内容と、未解決の品質上の限界を記録できる。（D5.O3）

## 前提知識

- D4で学んだ、測定定義、欠測、偏り、データ最小化。

## 導入: きれいに見える表が正しいとは限らない

架空の図書室座席調査を、次の表で受け取ったとします。

| row_id | date | time | people | available_seats |
| --- | --- | --- | ---: | ---: |
| R01 | 07-08 | 15:30 | 18 | 6 |
| R02 | 07/08 | 16:00 | 23 | 1 |
| R03 | 07-08 | 16:30 | 未記録 | 4 |
| R04 | 07-09 | 15:30 | 17人 | 7 |
| R04 | 07-09 | 15:30 | 17人 | 7 |
| R05 | 07-09 | 16:00 | 25 | -1 |

すべて合成データです。重複や不正値は、クリーニング判断を学ぶために意図的に含めています。

## 表の構造を読む

- **観測**: 一つの対象・時点について記録した一行。この例では各観測時点です。
- **変数**: 観測ごとに記録する項目。`time` や `people` などの列です。
- **値**: 行と列の交点にある個別の記録。`R02` の `people` は `23` です。
- **データ型**: 数値、文字列、日付、分類など、値の扱い方。
- **欠測**: 測る予定だったが値が得られなかった状態。0とは区別します。

`row_id` は観測を区別するキーとして使います。同じ `row_id` が二行あれば、重複か、キー設計の誤りかを確認します。

## クリーニング規則を先に書く

値を見ながら都合よく直すのではなく、規則と理由を先に決めます。

| 規則 | 適用例 | 注意 |
| --- | --- | --- |
| 日付形式を `YYYY-MM-DD` に統一 | `07/08` -> `2026-07-08` | 年が資料で確認できる場合だけ補う |
| 単位文字を分離 | `17人` -> `17` | 人数であることが列定義に明記されている場合 |
| 完全一致する重複を一件にする | 二つの `R04` | どちらを除いたか記録する |
| 不可能な値を欠測候補にする | 空席 `-1` | 正しい値を推測で補わない |
| 未記録を欠測として保持 | `未記録` -> 欠測 | 0へ置換しない |

異常値が本当に誤りとは限りません。特に混雑した日の人数は珍しくても有効かもしれません。元記録や測定範囲を確認できない場合は、削除せず「要確認」として残す選択もあります。

## 変換履歴を残す

再現可能にするには、結果だけでなく操作を記録します。

| step | target | rule | result | unresolved |
| ---: | --- | --- | --- | --- |
| 1 | `date` | 年を確認しISO形式へ統一 | 5行を変換 | なし |
| 2 | `people` | 末尾の単位「人」を分離 | R04を17へ変換 | なし |
| 3 | `row_id` | 全列一致の重複を一件にする | R04を一件除外 | 再測定かは確認不能 |
| 4 | `available_seats` | 0未満を要確認の欠測にする | R05を欠測 | 元値は不明 |

データ本体を上書きする場合でも、元データの読み取り専用コピーと変換履歴を分けて保存すると、判断を再確認できます。

## 分かち書き済みテキストを前処理する

文章も、規則を決めて語に分ければ度数を調べられます。ここでは形態素解析を使わず、教材側で `/` による分かち書きを済ませた合成コメントを使います。

| comment_id | 分かち書き済みの合成コメント |
| --- | --- |
| T01 | 図書室 / は / 静か / です / 。 |
| T02 | 静か / な / 席 / が / 便利 / です / 。 |
| T03 | 図書室 / の / 席 / は / 便利 / です / 。 |
| T04 | 案内 / が / 分かり / にくい / です / 。 |

分析前に、次の前処理規則を固定します。

1. `/` の左右の空白を取り除き、区切られた語を一つのトークンとする。
2. この例では表記を勝手に言い換えず、`分かり` と `にくい` も別のトークンとして保持する。
3. 頻度表では `は`、`が`、`の`、`な`、`です`、`。` を除外する。
4. 元のコメント表を読み取り専用で残し、除外語一覧、規則、実行順、結果を変換履歴へ記録する。

手作業で規則を適用すると、残る語は次の通りです。

| comment_id | 残すトークン |
| --- | --- |
| T01 | 図書室、静か |
| T02 | 静か、席、便利 |
| T03 | 図書室、席、便利 |
| T04 | 案内、分かり、にくい |

除外語や分割位置を変えると度数も変わります。日本語では語の境界が空白で明示されないため、分かち書きの規則は分析者の判断を含みます。元文を捨てると、「案内が分かりにくい」というまとまりや否定表現の文脈を復元できません。

実在する自由記述には、氏名、健康、家庭事情などが含まれる可能性があります。この活動では合成文だけを使い、実在する生徒のコメントを入力・共有させません。誰が回答できたか、短文だけを集めたかといった収集上の偏りも、前処理だけでは解消しません。

## Pythonで同じ規則を表す

次は合成データの一部を、Pythonで整える例です。ここで初めて使うPython固有の書き方を先に確認します。

- `{「列名」: 値}` の形をした**辞書**は、一行の値を列名で取り出すための入れ物です。`row["people"]` は、その行の `people` 列を読みます。
- `None` は、この例で欠測を表す目印です。0人とは異なります。
- `int(...)` は、数字として書かれた文字列を整数へ変換します。変換できる形式だと確認した値にだけ使います。
- `append(...)` は、クリーニング後の一行を結果のリスト末尾へ追加します。

これらの構文を暗記することが目標ではありません。重要なのは、規則を明示し、元の意味を推測で変えず、同じ処理を再実行できることです。

```python
rows = [
    {"row_id": "R01", "people": "18", "seats": "6"},
    {"row_id": "R02", "people": "23", "seats": "1"},
    {"row_id": "R03", "people": "", "seats": "4"},
]

cleaned = []
for row in rows:
    people = None
    if row["people"] != "":
        people = int(row["people"])
    cleaned.append([row["row_id"], people, int(row["seats"])])

print(cleaned)
```

実務では、表計算、データベース、専用ツールなど別の実装でも、同じ規則と履歴を保てます。

## よくある間違い

- 空欄をすべて0にする。未記録と0は意味が違います。
- 珍しい値を理由なく削除する。測定範囲と元記録を確認します。
- 表記を統一した後、何を変えたか残さない。
- 同じ値の行をすべて重複と考える。別の観測が偶然同じ値になることがあります。
- Pythonでエラーが出なければデータも正しいと考える。意味上の誤りは実行だけでは分かりません。
- 除外語と分割規則を記録せず、語だけの表から元文の意味を断定する。

## 自己確認

1. 一行、一列、一つの値を、観測、変数、値として説明できますか。
2. 欠測、0、異常値を区別できますか。
3. テキストの分割、除外、保持の規則と、失われる文脈を説明できますか。
4. 規則、対象、結果、未解決事項を変換履歴へ残せますか。

## 自己確認の診断と修正

- 列名だけを見て型を決めた場合は、値の範囲、単位、欠測表現を確認します。
- 「修正した」で止まった場合は、元値、規則、変更後、理由を記録します。
- 不正値を推測で埋めた場合は、元記録で確認できるか、欠測として残すべきかを再検討します。
- 語だけを残した場合は、読み取り専用の元文、分割規則、除外語一覧を保存します。

## まとめ

- 表形式データは、観測、変数、値、型、欠測、キーで構造を捉えます。
- クリーニング規則は適用前に明示し、意味を推測で変更しません。
- 元データ、変換履歴、未解決の品質上の限界を残すと、処理を再現・検証できます。
- テキスト分析では分割・除外の選択が結果と文脈を変えるため、元文と規則を残します。

## 別の場面への転移

表記揺れと欠測を含む小さな表、または分かち書き済みの短文一覧を紙で整えます。変更前に規則を決め、各変換を記録し、意味を変えるおそれや文脈が失われた箇所を未解決事項として残してください。
