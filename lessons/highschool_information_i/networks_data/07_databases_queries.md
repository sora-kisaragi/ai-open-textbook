# データベースと問い合わせ

## 学習目標

1. 構造化した保存が、一貫した検索と更新を支える理由を説明できる。（D8.O1）
2. 小さなデータベースのレコード、フィールド、キー、表、関係を識別できる。（D8.O2）
3. 絞り込み、列の選択、並べ替え、集約を行う単純な問い合わせを作り評価できる。（D8.O3）

## 前提知識

- D2で学んだ、情報サービスとデータ処理の流れ。
- D5で学んだ、表形式データ、型、欠測、キー。
- D6で学んだ、度数、割合、集約。

## 導入: 表を増やす目的は、情報をばらばらにすることではない

図書室サービスで、蔵書と分類名を一つの表へ毎回書くと、同じ分類名が何度も現れます。表記を一か所だけ直すと、残りが古いままになるかもしれません。データベースでは、意味に応じて構造を設計し、キーで関係を表します。

この章の蔵書、分類、貸出状況はすべて合成データです。実在する利用者や貸出履歴は扱いません。

## レコード、フィールド、キー、表

`books` 表を考えます。

| book_id | title | category_id | available |
| --- | --- | --- | --- |
| B201 | 星の観察入門 | C10 | 1 |
| B202 | データで見る天気 | C20 | 0 |
| B203 | はじめての統計 | C20 | 1 |

- 一行は一冊の**レコード**です。
- `title` などの列は**フィールド**です。
- `book_id` は各レコードを一意に識別する**主キー**です。
- `category_id` は別の表を参照する**外部キー**として使います。

`categories` 表は次です。

| category_id | category_name |
| --- | --- |
| C10 | 天文 |
| C20 | データ科学 |

`books.category_id` と `categories.category_id` を対応させると、一つの分類に複数の蔵書が属する関係を表せます。分類名は一か所で管理でき、表記の不一致を減らせます。

## 構造化が一貫性を支える

データベースは、単に大量のデータを置く箱ではありません。

- キーで同じ対象を区別する。
- 型や制約で許される値を限定する。
- 関係で重複を減らし、更新箇所を明確にする。
- 問い合わせで必要な行と列だけを取得する。
- 権限、履歴、バックアップなどの運用と組み合わせる。

構造があっても、誤った測定や偏ったデータが自動的に妥当になるわけではありません。D4-D7と同じ品質確認を続けます。

## 問いを操作へ分ける

問い合わせは、データへ尋ねる問いを操作として明示したものです。

| 操作 | 自然言語の問い | SQLの要素 |
| --- | --- | --- |
| 絞り込み | 貸出可能な本だけ | `WHERE available = 1` |
| 列の選択 | IDと題名だけ | `SELECT book_id, title` |
| 並べ替え | IDの昇順 | `ORDER BY book_id` |
| 集約 | 分類ごとの冊数 | `COUNT(*)` と `GROUP BY` |

貸出可能な本のIDと題名を並べます。

```sql
SELECT book_id, title
FROM books
WHERE available = 1
ORDER BY book_id;
```

分類ごとの冊数を数えます。

```sql
SELECT category_id, COUNT(*) AS book_count
FROM books
GROUP BY category_id
ORDER BY category_id;
```

SQLは特定の製品だけの考え方ではありませんが、細かな構文や機能はデータベース製品によって異なります。この教材ではオフラインのSQLiteで確認できる範囲の単純な問い合わせを扱います。

## 問い合わせ結果を評価する

SQLが実行できても、問いに答えているとは限りません。

- `available = 1` が「今すぐ貸出可能」の定義と一致するか。
- 欠測や不正値が含まれていないか。
- 並び順を指定しない結果へ順序を仮定していないか。
- `COUNT(*)` の対象が本当に一冊一行か。
- 必要のないフィールドまで取得していないか。

たとえば分類ごとの冊数を数えるとき、同じ本が重複行として二回登録されていれば結果も二重になります。問い合わせだけでなく、キーとデータ品質を確認します。

## 更新と整合性

更新では、「どのレコードを」「どの条件で」「何へ変えるか」を明確にします。条件を誤ると多くの行を変えるため、学習活動では元データのコピーを使い、対象行を検索してから変更案を確認します。

この教材では実在システムへ接続せず、個人データも使いません。許可のないデータベースを検索・変更してはいけません。

## よくある間違い

- 行番号を永続的な主キーと考える。並べ替えや追加で位置は変わります。
- 同じ名前なら同じ対象だと決める。一意なキーで対象を区別します。
- `SELECT *` で不要な情報まで常に取得する。
- `ORDER BY` なしの結果へ固定した順序を期待する。
- SQLが実行できれば、元データの品質や問いも正しいと考える。

## 自己確認

1. レコード、フィールド、主キー、外部キー、表の関係を説明できますか。
2. 自然言語の問いを、絞り込み、列選択、並べ替え、集約へ分けられますか。
3. 実行できた問い合わせに対しても、対象、欠測、重複、順序を確認できますか。

## 自己確認の診断と修正

- 表を分ける理由が「見やすさ」だけの場合は、重複更新と一貫性を考えます。
- SQLを暗記した場合は、各句が元の問いのどこに対応するか説明します。
- 結果だけを見た場合は、行の単位、キー、除外条件、並び順へ戻ります。

## まとめ

- データベースは、レコード、フィールド、キー、表、関係でデータを構造化します。
- 問い合わせは、絞り込み、列選択、並べ替え、集約として問いを明示します。
- 実行成功だけでなく、キー、データ品質、条件、結果の意味を評価します。

## 別の場面への転移

架空の備品台帳をカードで表し、レコード、フィールド、キー、二つの表の関係を示します。SQLを書かず、条件で絞る、列を選ぶ、並べ替える、件数をまとめる問い合わせを自然言語で作り、期待する結果を手作業で確かめてください。
