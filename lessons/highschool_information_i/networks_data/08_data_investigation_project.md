# データ調査プロジェクト

## 学習目標

1. 範囲を限定した問いを、倫理的で再現可能なデータ調査へ変えられる。（D9.O1）
2. データを収集または選択し、クリーニング、分析、可視化、記録を行える。（D9.O2）
3. 不確かさ、限界、感度分析、改善案を含めて結論を発表し、レビュー後に改訂できる。（D9.O3）

## 前提知識

- A6の比較・評価、B6の試作・利用者評価。
- D3のリスクと多層防御、D7の可視化と解釈、D8の構造化と問い合わせ。
- C2-C6のPythonは実装手段として利用できますが、特定の言語を使うこと自体は評価目的ではありません。

## プロジェクトの目的

架空の星見高校図書室について、**どの時間帯に座席利用が集中し、案内方法をどう改善できるか**を、提供された合成データで調べます。

このプロジェクトは、現実の学校運用を評価するものではありません。氏名、端末ID、貸出履歴などの個人データを集めません。実在する利用者への調査やオンライン公開は範囲外です。

## 成果物

次の十点を、一つの調査報告へまとめます。

1. 問い、対象範囲、判断に使う指標。
2. データの出所と、合成データであることの説明。
3. データ辞書、収集しない項目、読み取り専用で保存する元データの説明。
4. クリーニング規則と、日時・対象・操作・理由・結果を含む変換履歴。
5. 同じ入力から同じ結果を得られる集計手順または問い合わせと、手計算などによる照合。
6. 単位、欠測、標本範囲を示す表または可視化と、同じ内容の本文要約。
7. 証拠の範囲に限定した結論。
8. 変更した条件、変更前後の実結果、結論が変わったかを示す感度分析。
9. 不確かさ、限界、残余リスク、それらに対応する具体的な改善案。
10. 3分程度の発表または同等の読み取り可能な説明と、受けた指摘・判断・修正を残す改訂履歴。

見た目の完成度だけでなく、問いから証拠までの追跡可能性を評価します。

## 提供する合成データ

| record_id | day | time | people | available_seats | note |
| --- | --- | --- | ---: | ---: | --- |
| L01 | Mon | 15:30 | 12 | 12 |  |
| L02 | Mon | 16:00 | 21 | 3 |  |
| L03 | Mon | 16:30 | 16 | 8 |  |
| L04 | Tue | 15:30 | 14 | 10 |  |
| L05 | Tue | 16:00 | 19 | 5 |  |
| L06 | Tue | 16:30 | 未記録 | 7 | count interrupted |
| L07 | Wed | 15:30 | 13 | 11 |  |
| L08 | Wed | 16:00 | 24 | 0 |  |
| L09 | Wed | 16:30 | 18 | 6 |  |
| L10 | Wed | 16:30 | 18 | 6 | duplicate candidate |

定員は24席とする教材上の仮定です。`L09` と `L10` は識別子と注記を除く観測値が同じですが、重複か再観測かを元資料だけでは確定できません。どちらの扱いを選んだかと、その影響を報告します。

## 第1段階: 問いと境界を決める

問いを「図書室は混んでいるか」のままにせず、次のように限定します。

> 提供された月曜から水曜の観測では、空席が4席未満となる時間帯はいつか。その傾向を案内するなら、どの限界を併記すべきか。

この問いから、指標を `available_seats < 4` とします。別の基準を選ぶ場合は、分析前に理由を示します。

## 第2段階: データ辞書と品質確認

| 変数 | 意味 | 型 | 許される範囲・欠測 |
| --- | --- | --- | --- |
| `record_id` | 観測識別子 | 文字列 | 重複しない想定 |
| `day` | 観測曜日 | 分類 | Mon, Tue, Wed |
| `time` | 観測時刻 | 時刻 | 15:30, 16:00, 16:30 |
| `people` | 室内人数 | 整数 | 0〜24、欠測あり |
| `available_seats` | 利用可能席数 | 整数 | 0〜24 |

次を決め、変換履歴へ残します。

- `未記録` を0ではなく欠測として扱う。
- `L09` と `L10` を重複候補として感度分析する。
- 範囲外の値を推測で直さない。
- 元表を保存し、分析用の表を別に作る。

元表は読み取り専用の原本として保持し、分析用の表だけを変更します。原本を上書きした場合は、変換前後を第三者が照合できません。

## 第3段階: 集計を再現する

手計算、表計算、SQL、Pythonのどれでも構いません。Pythonを使う場合の最小例を示します。

```python
rows = [
    ["Mon", "15:30", 12, 12],
    ["Mon", "16:00", 21, 3],
    ["Mon", "16:30", 16, 8],
    ["Tue", "15:30", 14, 10],
    ["Tue", "16:00", 19, 5],
    ["Tue", "16:30", None, 7],
    ["Wed", "15:30", 13, 11],
    ["Wed", "16:00", 24, 0],
    ["Wed", "16:30", 18, 6],
]

crowded = []
for row in rows:
    if row[3] < 4:
        crowded.append([row[0], row[1], row[3]])

print(crowded)
```

この例は重複候補 `L10` を除外した分析です。除外しない結果も計算し、結論が変わるか確かめます。

## 第4段階: 可視化と解釈

最低限、次を含む表またはグラフを作ります。

- 曜日と時刻。
- 利用人数または空席数と単位。
- 欠測の表示。
- 混雑基準の説明。
- 合成データ、観測範囲、標本数の注記。

16:00の空席が3、5、0であるため、「提供された三日では16:00に空席が少ない」とは言えます。「毎週水曜16:00が必ず満席」「16:00という時刻が混雑の原因」とは言えません。

## 第5段階: 感度、限界、改善

分析判断を一つ変え、**変更前後の実際の結果**と、結論が変わったかを報告します。

- 混雑基準を4席未満から6席未満へ変える。
- `L10` を残す場合と除く場合を比べる。
- 利用人数ではなく空席数でまとめる。

たとえば混雑基準を4席未満から6席未満へ変えると、該当する観測は2件から3件へ増えます。16:00の3件がすべて6席未満になるため、「提供された三日では16:00に空席が少ない」という限定的な結論は変わりません。

ただし、該当件数と基準への依存は報告へ追加します。別の条件を選んだ場合も、変更前後の値と結論が変わったかを示します。

限界には、三日間だけ、各日三時点だけ、火曜16:30が欠測、重複候補を確定できない、座席不足が利用者の満足度を直接表すとは限らないことを含めます。

改善案は、結論を強く見せる案ではなく、証拠を良くする案にします。たとえば複数週を同じ時刻で観測する、観測中断の記録方法を決める、混雑基準を利用目的に照らしてレビューする、などです。

## 第6段階: 発表、相互レビュー、改訂

報告を3分程度で発表するか、同じ内容を見出し付き本文または読み上げ可能な音声原稿で提示します。グラフを使う場合も、傾向、比較値、欠測、標本範囲、結論を本文で説明します。色や画像だけに意味を持たせません。

相互レビューでは、個人の経験や未収集情報を求めず、次の問いだけを使います。

- 問いと指標は対応しているか。
- 元データから結果を再実行できるか。
- 表や可視化と本文要約は同じ内容か。
- 感度分析に変更前後の実結果があるか。
- 結論、限界、残余リスク、改善案は対応しているか。

改訂履歴には、版、日付、受けた指摘、採用・不採用の判断、変更箇所と理由を残します。指摘をすべて採用することではなく、判断を説明し、報告を再検証できることが目的です。

## 4〜5時限の進行例

1. 第1時: 問い、範囲、指標、データ辞書を決める。
2. 第2時: 品質確認、クリーニング、変換履歴を作る。
3. 第3時: 集計、問い合わせ、再実行確認を行う。
4. 第4時: 可視化と本文要約、結論、感度分析、限界、改善案をまとめる。
5. 第5時: 発表または同等の説明、相互レビュー、改訂履歴を伴う修正を行う。

## よくある間違い

- グラフを作ってから問いを決める。
- 欠測を0人として扱う。
- 重複候補を説明なく削除する。
- 16:00との関連を、時刻が混雑を起こした因果関係として書く。
- 合成データの結果を実在の図書室へ一般化する。
- Pythonが動いたことだけを再現性の証拠にする。
- 感度分析で条件名だけを書き、変更前後の値や結論の変化を示さない。
- グラフだけを提出し、同じ内容の本文要約や改訂履歴を残さない。

## 自己確認

1. 問い、指標、対象範囲が一貫していますか。
2. 元データから結論まで、変換と計算を再実行できますか。
3. 感度分析の実結果、言えること、言えないこと、残余リスク、次の改善を分けていますか。
4. 表現の本文代替、発表内容、レビュー後の改訂履歴がありますか。

## 自己確認の診断と修正

- 問いが広い場合は、期間、対象、指標を限定します。
- 結果が再現できない場合は、元表、除外規則、計算式、問い合わせをそろえます。
- 結論が強すぎる場合は、観測範囲、欠測、偏り、代替説明へ戻ります。
- 感度分析が操作名だけの場合は、変更前後の値と結論が変わったかを追記します。
- レビュー後の完成版だけがある場合は、指摘、判断、変更理由を改訂履歴へ戻します。

## まとめ

- データ調査は、問い、倫理、出所、変換、分析、伝達を一つの追跡可能な流れにします。
- PythonやSQLは再実行の手段であり、問いやデータの意味を自動的に保証しません。
- 結論は証拠の範囲に限定し、不確かさ、限界、残余リスク、改善案を明示します。
- 発表と相互レビューを経て、本文代替と改訂履歴を含む再検証可能な成果物にします。
