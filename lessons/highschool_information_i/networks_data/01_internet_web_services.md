# インターネット、Web、情報サービス

## 学習目標

1. インターネット、Web、情報サービスと、URI・URLの役割を区別できる。（D2.O1）
2. URL入力からDNS、経路制御、伝送制御、TLS、HTTP、処理、画面表示までの流れをたどれる。（D2.O2）
3. 観察できる事実から、失敗した層や構成要素を絞り、サービスへの影響を評価できる。（D2.O3）

## 前提知識

- D1で学んだ、クライアントとサーバー、LANとWAN、スイッチとルーター、IPアドレス、プロトコルの層。

## 導入: アドレスを入力した後に起こること

ブラウザーへURLを入力すると、すぐにページが見えることがあります。しかし、端末はURLをそのままネットワークへ投げているわけではありません。名前をIPアドレスへ対応付け、複数のネットワークを越え、安全な通信路を準備し、Webサーバーへ要求を送ります。

この章では、架空の図書検索URLを一つの手掛かりとして、利用者の操作から画面表示までを追跡します。

## インターネット、Web、情報サービスを区別する

| 用語 | この教材での意味 | 図書検索での例 |
| --- | --- | --- |
| インターネット | 複数のネットワークをIPなどの共通規則で相互接続した基盤 | パケットを学校のLANから遠隔のサーバーへ運ぶ |
| Web | この教材では、ブラウザーがURIで対象を示し、HTTPの要求と応答で表現を受け取る仕組みとして扱う | ブラウザーが検索ページを要求する |
| 情報サービス | 利用者の目的を満たすため、データ、処理、機器、人、運用を組み合わせた働き | 蔵書を検索し、貸出可否を示す |

Webはインターネットを利用する仕組みの一つです。電子メールなど、インターネットを使ってもWebそのものではない仕組みがあります。また、情報サービスは画面だけではなく、サーバー、データベース、更新手順、障害対応などを含みます。

## URIとURLを読む

**URI**は資源を識別する文字列の総称です。この教材では、`https` の方式と接続先を含むURIを、一般的な呼び方に合わせて**URL**と呼びます。次の文字列はURIであり、この教材で扱うURLでもあります。

`https://library.example.test/books/B204?view=summary`

| 部分 | 値 | 役割 |
| --- | --- | --- |
| スキーム | `https` | HTTPSでアクセスすることを示す |
| ホスト名 | `library.example.test` | 接続先を名前で示す |
| パス | `/books/B204` | サーバー内で要求する資源を示す |
| クエリ | `view=summary` | 表示方法など、要求に追加する情報を示す |

`.test` は例示用の名前です。教材外の実在サービスへの接続には使いません。

## URLから画面表示までをたどる

教材では `library.example.test` が文書用IPアドレス `192.0.2.24` に対応するものとします。どちらも実在する接続先ではありません。

1. **URLの解釈**: ブラウザーがスキーム、ホスト名、パス、クエリを読み分ける。
2. **DNSによる名前解決**: 端末がDNSへホスト名を問い合わせ、対応するIPアドレス `192.0.2.24` を得る。
3. **リンクと経路制御**: LANのスイッチがパケットをルーターへ渡し、複数のルーターが宛先IPアドレスに応じて次のネットワークへ転送する。
4. **伝送制御**: トランスポート層がクライアントとサーバーの間で通信路を準備する。この教材では、順序や欠落を扱うTCPを例にする。
5. **TLSによる安全な通信路**: サーバー証明書を検証し、通信相手と暗号化に使う情報を確認して、盗み見や改ざんを抑える通信路を作る。
6. **HTTP要求**: ブラウザーがパスとクエリを含む要求をWebサーバーへ送る。
7. **サービス内の処理**: Webサーバーが要求を検査し、アプリケーションがデータベースから蔵書情報を調べる。
8. **HTTP応答**: サーバーが状態と結果を返す。
9. **表示**: ブラウザーが応答を解釈し、見出しや表として利用者へ示す。

この九段階は、二つの通信をまとめた表現です。最初にDNSサーバーへ問い合わせる通信があり、その通信自体もD1で学んだリンク、IP、トランスポートの層を使います。名前解決後、得られたIPアドレスを宛先としてWebサービスへの別の通信を始めます。

**HTTPS**は、HTTPの要求と応答をTLSで保護して運ぶ仕組みです。TLSは通信路上の機密性、完全性、サーバー認証を支えますが、ページに書かれた内容が正しいこと、接続先が信頼できる事業者であること、端末が安全であることまでは保証しません。

実際の通信にはキャッシュ、複数のDNSサーバー、負荷分散、複数の通信方式などが加わります。ここでは、障害箇所を考えられる最小限の経路へ単純化しています。

## どの層・構成要素で失敗したか

| 観察 | 候補となる層・構成要素 | 根拠を強める次の確認 |
| --- | --- | --- |
| ホスト名では開けないが、教材で与えたIPアドレスへの到達確認は成功する | DNS | 別の既知の名前は解決できるか |
| 同じLAN内は使えるが、外部の複数サービスへ届かない | ルーターまたは経路制御 | ルーターまでの通信、別端末との差 |
| DNSは成功するが、端末間の通信路を準備できない | トランスポートまたはサーバー待受け | 別のサービスや時間帯との比較 |
| 証明書の名前がURLのホスト名と一致しないと表示される | TLS・証明書検証 | 接続を止めて担当者へ報告し、URLの入力と警告の種類を確認する |
| ページは開くが検索要求だけ失敗する | HTTP要求、アプリケーション、データベース | 応答状態、要求形式、既知の蔵書IDとの差 |
| 結果が古い | キャッシュまたはデータ更新 | 更新時刻、参照したデータ、保持時間 |

「開かない」という見た目だけで原因を断定しません。どの段階まで成功したか、同じ条件で何が成功するかを比較し、一度に一条件ずつ確かめます。実在するネットワークを許可なく調査せず、教材内の観察記録だけを使います。

## 情報システムをサービスとして評価する

サービスが技術的に応答しても、利用者の目的を十分に満たすとは限りません。

- **有効性**: 目的に合う情報を誤りなく得られるか。
- **利用可能性**: 必要な時間と場所で使えるか。
- **性能**: 許容できる時間で応答するか。
- **アクセシビリティ**: キーボード、読み上げ、文字拡大などで利用できるか。
- **運用性**: 更新、証明書管理、障害対応、問い合わせの担当が決まっているか。
- **安全性**: 権限のない閲覧や変更、通信路上の盗み見や改ざんを抑えられるか。

たとえば応答を速くするために結果を長くキャッシュすると、性能は上がっても最新性が下がる場合があります。評価では一つの指標だけで結論を出さず、誰のどの必要を満たすか、どの層や運用へ影響するかを示します。

## よくある間違い

- インターネットとWebを同じ意味で使う。
- URL全体をDNSがIPアドレスへ変換すると考える。DNSが対応付ける中心はホスト名です。
- DNSが検索結果を作ると考える。DNSは名前解決、検索結果は情報サービス内の処理です。
- HTTPSならページ内容や運営者の行動まで正しいと考える。TLSが保護する範囲と、内容の信頼性を分けます。
- 失敗原因をすべてネットワークとする。TLS、HTTP、アプリケーション、データにも原因があり得ます。

## 自己確認

1. URLのスキーム、ホスト名、パス、クエリを読み分けられますか。
2. DNS、経路制御、伝送制御、TLS、HTTP、サービス内処理を順に説明できますか。
3. 証明書警告と検索結果なしでは、疑う層が異なる理由を説明できますか。

## 自己確認の診断と修正

- URL全体を一語で扱った場合は、ホスト名と、サーバーへ送るパス・クエリを分けます。
- 流れが「接続して表示」で終わった場合は、DNS、ルーター、トランスポート、TLS、HTTPを間へ補います。
- 障害原因を一つに決めた場合は、その候補なら何が成功し、何が失敗するかを一つずつ書きます。

## まとめ

- インターネットは接続基盤、WebはURIとHTTPなどで資源を扱う仕組み、情報サービスは利用目的を満たす全体です。
- URLを分解すると、DNSが扱うホスト名と、HTTP要求が扱うパスやクエリを区別できます。
- 名前解決、経路制御、伝送制御、TLS、HTTP、サービス内処理へ分けると、通信経路と障害箇所を説明できます。
- HTTPSは通信路を保護しますが、内容の正しさやすべての安全性を保証するものではありません。
