# ネットワークとプロトコル

## 学習目標

1. 小規模なネットワークの端末、伝送路、中継機器、プロトコルを識別できる。（D1.O1）
2. 通信に共有された規則が必要な理由を説明できる。（D1.O2）
3. 通信目的と信頼性の条件から、小規模なネットワークを設計できる。（D1.O3）

## 前提知識

- B1で学んだ、目的と相手に応じて通信手段を選ぶ考え方。
- C1で学んだ、入力・記憶・処理・出力と、装置間を流れるデータ。

## 導入: 1台ではできないこと

架空の星見高校では、図書室の貸出状況を閲覧する端末と、データを管理するサーバーをつなぎます。1台の端末だけなら、その端末の中で処理が完結します。複数の端末が同じ情報サービスを使うには、データの通り道と、送り方の規則をそろえます。

この章の学校名、機器名、アドレス、記録はすべて教材用の合成事例です。実在する学校のネットワーク構成ではありません。

## ネットワークを構成する役割

ネットワークは、複数の機器が情報を送受信できるように接続された仕組みです。機器名ではなく役割で捉えます。

| 役割 | 合成事例 |
| --- | --- |
| 端末（エンドポイント） | 閲覧端末、管理用PC、サーバー |
| 伝送路（リンク） | 有線LAN、無線LANなど、データが通る区間 |
| 中継機器 | 複数の伝送路をつなぎ、データを次の区間へ渡す機器 |
| プロトコル | 形式、順序、応答など、通信相手が共有する規則 |

一つの機器が複数の役割を持つ場合もあります。重要なのは、どこが送信元と宛先で、どの区間と規則を使ってデータが進むかです。

## データを小さな単位で運ぶ

ネットワークでは、送る情報を一定の単位に分けて運ぶことがあります。この教材では、その単位を**パケット**と呼びます。パケットには、内容の一部だけでなく、宛先や順序を扱うための情報も含めます。

図書室端末がサーバーへ検索語を送る流れを単純化すると、次のようになります。

1. 端末が検索要求を決められた形式にする。
2. 要求をパケットとして伝送路へ送る。
3. 中継機器が宛先に近い次の区間へ渡す。
4. サーバーが要求を受け取り、規則に従って応答する。
5. 端末が応答を解釈して画面に表示する。

実際の通信では複数のプロトコルが層のように協力します。この章では製品固有の設定値を暗記せず、「各規則がどの役割を受け持つか」を考えます。

## なぜ共有された規則が必要か

同じ伝送路につながっていても、送信側と受信側が次を共有していなければ通信は成立しません。

- どの順序で情報を並べるか。
- どこまでが一つのメッセージか。
- 誰宛ての情報か。
- 正常に受け取れなかったとき、どう扱うか。
- 応答をいつまで待つか。

たとえば、送信側がページ数を「6」、受信側が「文字コードの番号6」と解釈すれば、ビット列が届いても意味は一致しません。通信が成功するには、物理的に届き、同じ規則で意味を解釈できるという二つの条件を満たします。

## 要件から小規模ネットワークを設計する

次の合成要件を考えます。

- 閲覧端末2台から図書検索を利用する。
- 管理用PCだけが蔵書情報を更新できる。
- 1本の伝送路が使えないときも、管理用PCからサーバーへ連絡できる案を検討する。
- 個人情報は扱わず、蔵書IDと貸出可否だけを使う。

設計では、まず必須の経路を文章で表します。

`閲覧端末 -> 中継機器 -> サーバー`

次に、信頼性を高める案を比較します。予備の伝送路を追加すれば、ある区間の故障に備えられますが、費用、設定、保守の負担は増えます。すべてを二重化するのではなく、停止の影響と費用を比べて決めます。

設計を説明するときは、機器の個数だけでなく、次を残します。

1. 誰と誰が通信するか。
2. 何を送るか。
3. どの経路を使うか。
4. どの規則が必要か。
5. 故障時に何が止まり、どこまで代替できるか。

## よくある間違い

- インターネットだけをネットワークと考える。校内だけの接続もネットワークです。
- 中継機器をデータの最終的な利用者と考える。中継と端末の役割を分けます。
- パケット一つが必ず完全な文書一つに対応すると考える。分割や再構成が行われる場合があります。
- 接続できれば意味も通じると考える。形式や順序などの共有規則もそろえます。
- 予備経路があれば必ず無停止になると考える。端末、電源、設定など別の障害点は残ります。

## 自己確認

1. 端末、伝送路、中継機器、プロトコルを、図書検索の流れへ対応付けられますか。
2. 「データは届いたが通信は成功していない」例を説明できますか。
3. 予備経路を追加しても残る障害点を一つ挙げられますか。

## 自己確認の診断と修正

- 機器名だけになった場合は、送信元、宛先、中継、規則の役割を書き添えます。
- 「同じ規則が必要」で止まった場合は、形式、順序、宛先、応答のどれがずれるかを具体化します。
- 信頼性を「機器を増やすこと」とだけ書いた場合は、守りたい通信と残る単一障害点を確認します。

## まとめ

- ネットワークは端末、伝送路、中継機器と、共有された通信規則で成り立ちます。
- パケットが届くことと、同じ意味として解釈できることは別の条件です。
- 設計では通信目的、経路、規則、障害時の影響と費用を対応付けます。

## 練習

- **共有規則が必要な理由を説明する**
- **通信要件から構成を選ぶ**
- **小規模ネットワークを改善する**
